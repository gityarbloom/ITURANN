# הצעת שיפור לפיצ'ר הניטור

## רקע

כיום Grafana
קוראת את הדשבורדים מקובצי 
JSON המאוחסנים ב-Bucket, 
 ומבצעת שאילתות ישירות על לוגים המאוחסנים ב-Cloud Logging.

לגישה זו מספר מגבלות: - הלוגים נשמרים למשך כ-30 יום בלבד. - יכולות
הסינון והשאילתות על JSON מוגבלות. - ניתוח תקלות היסטוריות הופך לקשה ככל
שהמידע נמחק. - קשה לבצע ניתוחים מורכבים או להוסיף אינדקסים.

## הארכיטקטורה המוצעת

בנוסף לקונטיינר הקיים של Grafana יתווספו:

1.  קונטיינר Python.
2.  קונטיינר MySQL.
3.  Persistent Volume עבור MySQL.

Terraform ימשיך לנהל את כלל התשתית: - יצירת VM. - ניהול Secrets. - הפעלת
Docker. - משיכת כלל ה-Images מ-Artifact Registry. - הרצת Grafana, Python
ו-MySQL.

### תהליך העבודה

1.  Python שולף לוגים מ-Cloud Logging.
2.  מבצע סינון, ניקוי והעשרה.
3.  שומר את המידע במסד הנתונים.
4.  Grafana קוראת את הנתונים ישירות מ-MySQL.

## יתרונות

### שמירת מידע לטווח ארוך

-   ללא תלות במגבלת 30 הימים של Cloud Logging.
-   היסטוריה מלאה לצורך חקירה והשוואה.

### יכולות סינון מתקדמות

-   שאילתות SQL.
-   חיפוש לפי מזהים, סטטוסים, זמנים ושדות נוספים.
-   אינדקסים לשיפור ביצועים.

### ניטור איכותי יותר

-   דשבורדים עשירים יותר.
-   הצגת מגמות לאורך חודשים.
-   מדדים מצטברים וזיהוי חריגות.

### ניפוי תקלות

-   חיפוש מהיר של אירועים.
-   הצלבת נתונים ממספר מקורות.
-   זיהוי נקודות כשל ותבניות חוזרות.

### אמינות ועמידות

-   Persistent Volume מגן על הנתונים גם במקרה של יצירה מחדש של
    הקונטיינרים.
-   Terraform ממשיך לספק תהליך פריסה אחיד וניתן לשחזור.

## סיכום

הפתרון מוסיף שכבת עיבוד ואחסון ייעודית ללוגים, ומשפר משמעותית את יכולות
הניטור, החקירה והתחזוקה של המערכת, תוך שמירה על ארכיטקטורה פשוטה המבוססת
על Docker ו-Terraform.
